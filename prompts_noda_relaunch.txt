┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #1 — [CRITICAL-1] Исправить genesis/faucet ownership
│ Статус: [СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас в проекте genesis supply 11,000,000 зачисляется на захардкоженный
block.GenesisAddress, а faucet-кошелёк берётся из любого переданного FAUCET_KEY.
Из-за этого нода может показывать faucet_active=true, но реальный баланс faucet
равен нулю, и /faucet падает с insufficient balance. Для публичного запуска это
критический дефект инициализации цепи.

ЗАДАЧА: Сделать так, чтобы владелец genesis/faucet определялся корректно и
детерминированно, а запуск с несовместимым ключом был невозможен.

Конкретные шаги:
1. Убери зависимость от захардкоженного block.GenesisAddress как единственного
   владельца 11M genesis supply.
2. На первом запуске новой сети:
   а) либо используй адрес, выведенный из GENESIS_PRIVATE_KEY / FAUCET_KEY,
   б) либо явно введи отдельный GENESIS_ADDRESS/GENESIS_PUBKEY и проверь, что
      приватный ключ действительно ему соответствует.
3. Зафиксируй выбранный genesis owner прямо в genesis block / chain metadata,
   чтобы на следующих стартах владелец уже не вычислялся «по-новому».
4. Если нода стартует на существующих данных и передан другой faucet key:
   - НЕ молча продолжай работу;
   - выдай fail-fast ошибку с понятным текстом о несовместимом faucet owner.
5. Добавь безопасную миграцию legacy-состояния:
   - если chain высоты 0 и faucet ещё ничего не раздал, разреши одноразовую
     миграцию старого hardcoded genesis owner на новый конфигурируемый owner;
   - если сеть уже ушла дальше genesis, запрещай опасную переинициализацию.
6. Обнови /status так, чтобы он явно показывал:
   - genesis_owner
   - faucet_owner
   - совпадают ли они
   - usable_faucet_balance
7. Добавь тесты:
   - новый случайный ключ действительно получает контроль над genesis/faucet;
   - несовместимый ключ на повторном старте приводит к понятной ошибке;
   - legacy height=0 мигрируется безопасно.
8. Выполни:
   go build ./...
   go test ./...
   go vet ./...
9. Зафиксируй одним коммитом:
   fix(consensus): bind genesis supply to configured faucet owner and reject mismatched keys [CRITICAL-1]
10. Смержи в main.

Критерий успеха: faucet работает с реальным балансом на свежем запуске, а узел
не может притворяться «активным faucet», если ключ не контролирует genesis funds.


┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #2 — [CRITICAL-2] Перевести консенсус на настоящие tx inputs/outputs
│ Статус: [НЕ СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас block.Transaction по сути account-like: From / To / Amount / Signature.
UTXO-слой лишь «симулирует» настоящую модель, сам выбирая UTXO отправителя и
создавая change output поверх упрощённой структуры. Для реальной монеты этого
недостаточно: консенсус должен видеть явные inputs и outputs.

ЗАДАЧА: Переделать транзакционный формат на полноценную UTXO-модель.

Конкретные шаги:
1. Замени текущую consensus-структуру Transaction на модель вида:
   - Version
   - Inputs []TxInput
   - Outputs []TxOutput
   - LockTime (можно минимально поддержать)
   - ID / Hash
2. Введи TxInput минимум с полями:
   - PrevTxID
   - PrevIndex
   - Signature
   - PubKey / WitnessData (минимально необходимое для проверки)
3. Введи TxOutput минимум с полями:
   - Amount
   - ScriptPubKey / Address
4. Реализуй deterministic tx serialization и txid/hash НЕ через fmt.Sprintf,
   а через стабильное бинарное/каноническое кодирование.
5. Реализуй sighash/signature verification по inputs/outputs, чтобы подпись
   зависела от конкретных prevouts и outputs, а не от строки From:To:Amount.
6. Перепиши utxo.ApplyBlock и все UTXO-операции:
   - больше НЕ искать «все UTXO отправителя по адресу» внутри консенсуса;
   - тратить ровно те outpoints, которые указаны во входах транзакции;
   - change output должен явно присутствовать в outputs транзакции.
7. Сохрани удобный wallet-level builder для API/CLI, который может собрать
   inputs/outputs автоматически, НО консенсусный блок должен хранить именно
   явные inputs/outputs.
8. Для старого формата продумай миграцию:
   - либо legacy chains читаются через конвертер;
   - либо узел явно отказывается загружать legacy data и пишет понятную
     инструкцию миграции. Молчаливый unsafe fallback запрещён.
9. Обнови тесты блока, UTXO, mempool и integration на новый формат.
10. Выполни:
    go build ./...
    go test ./...
    go vet ./...
11. Зафиксируй одним коммитом:
    refactor(consensus): replace account-like tx model with explicit UTXO inputs and outputs [CRITICAL-2]
12. Смержи в main.

Критерий успеха: внутри блоков больше нет консенсусной логики вида From/To/Amount;
каждая регулярная транзакция явно тратит prevouts и явно создаёт outputs.


┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #3 — [CRITICAL-3] Отвязать майнинг от отправки транзакции
│ Статус: [НЕ СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас SubmitTransaction кладёт tx в mempool и почти сразу майнит блок с этой
одной транзакцией. Это превращает сеть в «автоподтверждаемую демку», а не в
самостоятельную монету с mempool, блок-билдером, fees и отдельным майнером.

ЗАДАЧА: Сделать нормальный pipeline: mempool → block template → mining worker.

Конкретные шаги:
1. SubmitTransaction должен:
   - валидировать tx;
   - класть tx в mempool;
   - возвращать txid и статус pending;
   - НЕ майнить блок синхронно.
2. Создай отдельный miner service / package с background loop:
   - собирает block template из mempool;
   - добавляет coinbase;
   - включает несколько транзакций, а не одну;
   - работает через context для graceful shutdown.
3. Введи transaction fee:
   - fee = sum(inputs) - sum(outputs);
   - запрещай отрицательную fee;
   - coinbase забирает block reward + total fees блока.
4. Измени mempool ordering:
   - сортировка по fee rate (или fee, если проще на первом этапе),
   - при равенстве — FIFO.
5. Добавь конфиг майнинга:
   - MINING_ENABLED
   - MINER_ADDRESS
   - BLOCK_MAX_TX
   - MINING_MAX_ATTEMPTS / target behaviour
6. Добавь API/статус:
   - mempool size
   - mining enabled
   - miner address
   - last mined block hash
   - pending tx confirmations=0 до включения в блок
7. Обнови /send и /transaction ответы: они должны возвращать txid и pending,
   а не делать вид, что transaction уже подтверждена блоком.
8. Добавь тесты:
   - tx попадает в mempool и не подтверждается мгновенно;
   - miner забирает несколько tx в один блок;
   - fees корректно уходят в coinbase.
9. Выполни:
   go build ./...
   go test ./...
   go vet ./...
10. Зафиксируй одним коммитом:
    feat(miner): decouple mining from submission and build blocks from mempool with fees [CRITICAL-3]
11. Смержи в main.

Критерий успеха: пользовательская транзакция после отправки сначала живёт в
mempool, а подтверждение появляется только после работы отдельного майнера.


┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #4 — [CRITICAL-4] Лучшая цепь по cumulative work + reorg/orphans
│ Статус: [НЕ СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас сеть ориентируется на longest chain / replace if longer. Для настоящего
PoW-узла этого недостаточно. Нужны cumulative work, fork point, reorg, orphan
pool и корректный rollback/apply chainstate.

ЗАДАЧА: Реализовать полноценный best-chain selection по cumulative work.

Конкретные шаги:
1. Добавь block index:
   - hash -> block metadata
   - parent hash
   - height
   - cumulative work
   - статус: main/orphan/side
2. Реализуй расчёт cumulative work для каждого блока и выбор best tip именно по
   суммарной работе, а не по длине.
3. Реализуй orphan pool:
   - блоки без известного родителя временно сохраняются;
   - когда приходит родитель, orphan повторно обрабатывается.
4. Реализуй reorg pipeline:
   - найти fork point;
   - rollback UTXO/chainstate до fork point;
   - применить новую лучшую ветку;
   - вернуть в mempool транзакции из отсоединённых блоков, если они всё ещё валидны.
5. Убери опасный ReplaceChain(newChain) «целиком и сразу», если он сравнивает
   лишь длину. Вместо этого обработка должна идти через block index и best-tip.
6. Расширь P2P sync:
   - headers-first sync минимум в базовом виде;
   - сначала headers/metadata, потом block bodies;
   - не скачивать слепо полную цепь JSON-объектом.
7. Добавь integration tests:
   - две конкурирующие ветки разной длины и разной cumulative work;
   - reorg откатывает UTXO корректно;
   - orphan block подключается после прихода родителя.
8. Выполни:
   go build ./...
   go test ./...
   go vet ./...
9. Зафиксируй одним коммитом:
   feat(consensus): add cumulative-work chain selection with reorg and orphan handling [CRITICAL-4]
10. Смержи в main.

Критерий успеха: узел выбирает лучшую цепь по total work, умеет переживать
fork/reorg и не зависит от наивного «кто длиннее — тот прав».

│ ПРОМТ #5 — [CRITICAL-5] Убрать private_key из HTTP API в production
│ Статус: [НЕ СДЕЛАН]

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас /sign и /send принимают private_key через HTTP. Для публичного запуска это
недопустимо: full node не должен становиться custodial hot-wallet и принимать
секреты пользователей по сети.

ЗАДАЧА: Удалить опасную wallet-логику из production API и вынести её в offline CLI.

Конкретные шаги:

   ALLOW_INSECURE_WALLET_HTTP=false по умолчанию.
3. Если insecure mode выключен:
   - /sign и /send должны отвечать 403/410 с понятным текстом,
   - README должен прямо объяснять, что node API не принимает private keys.
4. Добавь CLI-команды или отдельный wallet package:
   - wallet new
   - wallet build-tx
   - wallet sign-tx
   - wallet send-raw / broadcast
5. Основной production endpoint должен принимать только raw signed transaction
   (или новый canonical tx JSON/hex), без секретных ключей.
6. Убедись, что faucet/admin-ключи не логируются и не возвращаются клиенту.
7. Обнови README, примеры curl и Docker docs под новую безопасную модель.
8. Добавь тесты:
   - insecure endpoints отключены по умолчанию;
   - raw signed transaction проходит;
   - старые /send /sign работают только в явно включённом dev-mode.
9. Выполни:
   go build ./...
   go test ./...
   go vet ./...
10. Зафиксируй одним коммитом:
    fix(api): remove private-key wallet operations from production HTTP API [CRITICAL-5]
11. Смержи в main.

Критерий успеха: публичный node больше не принимает приватные ключи через HTTP,
а подпись транзакций происходит оффлайн/локально у пользователя.


┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #6 — [HIGH-1] Перевести хранение с monolithic JSON на blockstore
│ Статус: [НЕ СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас ledger сохраняется одним JSON-файлом через os.WriteFile. Для публичной
сети это слишком хрупко: плохая масштабируемость, риск потери данных при крэше,
медленный full rewrite на каждый апдейт, отсутствие block index/chainstate store.

ЗАДАЧА: Сделать crash-safe persistent storage, пригодный для реального узла.

Конкретные шаги:
1. Раздели хранение минимум на:
   - blockstore
   - chainstate/UTXO state
   - node metadata (best tip, target, total mined, total faucet, genesis owner)
   - peers/banlist при необходимости
2. Перестань сериализовать весь Ledger целиком на каждое изменение.
3. Реализуй atomic write pattern:
   - write temp file
   - fsync/safe flush где возможно
   - rename поверх старого
4. При старте узла добавь recovery:
   - если найден temp/journal после падения, корректно восстановиться;
   - если chainstate повреждён, уметь rebuild из blockstore.
5. Добавь block index для быстрого доступа по hash/height.
6. Сохрани совместимость запуска:
   - при наличии старого node_data.json выполни одноразовую миграцию в новый
     формат и пометь её завершённой;
   - после успешной миграции старый файл не должен оставаться source of truth.
7. Добавь тесты на:
   - restart persistence;
   - recovery после частично записанного состояния;
   - rebuild UTXO из blockstore.
8. Выполни:
   go build ./...
   go test ./...
   go vet ./...
9. Зафиксируй одним коммитом:
   feat(storage): introduce crash-safe blockstore and chainstate persistence [HIGH-1]
10. Смержи в main.

Критерий успеха: узел больше не зависит от одного общего JSON-файла и умеет
корректно восстанавливаться после перезапуска/частичного сбоя записи.


┌─────────────────────────────────────────────────────────────────────────────
│ ПРОМТ #7 — [HIGH-2] Усилить P2P-протокол и анти-DoS защиту
│ Статус: [НЕ СДЕЛАН]
└─────────────────────────────────────────────────────────────────────────────

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

Сейчас framing уже есть, но для launch-ready монеты нужно сильнее защитить P2P:
валидация payload, checksums, self-connection detection, peer reputation,
anti-spam limits, inbound quotas и более строгая обработка malformed peers.

ЗАДАЧА: Довести P2P до состояния «не развалится от первого мусорного соседа».

Конкретные шаги:
1. Добавь checksum в header/message framing и проверку целостности payload.
2. Добавь self-connection detection по node nonce/node ID, чтобы узел не строил
   петли сам с собой.
3. Введи per-peer limits:
   - max inv items
   - max addr items
   - max outstanding getdata
   - rate limit по сообщениям/сек
4. Усиль ban score policy:
   - bad checksum
   - invalid payload schema
   - oversized inv/addr flood
   - repeated malformed tx/block announcements
5. Сделай более строгую валидацию peer addresses:
   - не принимать мусорные/невалидные адреса,
   - не сохранять loopback/private addrs как публичные seeds без явного allowlist.
6. Убедись, что входящие tx/block проходят полноценную валидацию ДО relay дальше.
7. Ограничь fan-out и дублирование relay, чтобы не спамить сеть одинаковыми inv.
8. Добавь unit/integration tests на malformed messages и peer banning.
9. Выполни:
   go build ./...
   go test ./...
   go vet ./...
10. Зафиксируй одним коммитом:
    feat(p2p): harden protocol with checksums, anti-spam limits and stricter peer validation [HIGH-2]
11. Смержи в main.

Критерий успеха: один плохой peer не может легко завалить ноду мусорными
сообщениями или заставить её ретранслировать невалидные данные.

Ты работаешь с репозиторием Bihan293/Noda (Go 1.22).

добавлять «ещё фичу», а доказать, что монету можно запускать без стыда.

ЗАДАЧА: Собрать полный финальный quality gate для публичного релиза.

Конкретные шаги:
1. Добавь/обнови integration tests для сценариев:
   - faucet bootstrap и exhaust до 11M;
   - mining rewards суммарно не превышают 10M;
   - total supply никогда не превышает 21M;
   - tx lifecycle: build → sign → mempool → mine → confirm;
   - restart persistence;
   - competing branches + reorg;
   - multi-node sync после временного расхождения.
2. Добавь property/invariant tests где уместно:
   - sum(UTXO) == genesis + mined - provably burned (если burn нет, то просто равен total supply);
   - no double-spend across main chain;
   - faucet never distributes above cap.
3. Запусти и зафиксируй race/fuzz/lint pipeline:
   - go test -race ./...
   - go test -fuzz=Fuzz -fuzztime=10s ./...   (или минимально по parser/serializer пакетам)
   - go vet ./...
4. Усиль GitHub Actions так, чтобы CI падал на любой из этих проверок.
5. Обнови README/ROADMAP/CHANGELOG:
   - убери устаревшие обещания;
   - опиши текущую архитектуру честно;
   - опиши безопасный production flow без передачи private_key в node API;
   - добавь раздел Known limitations, если что-то ещё осталось вне scope.
6. Если после всех шагов остаются недоделанные launch-blockers — НЕ прячь их,
   а явно перечисли в README как release blockers.
7. Выполни:
   go build ./...
   go test ./...
   go test -race ./...
   go vet ./...
8. Зафиксируй одним коммитом:
   chore(release): add final launch gate with invariants, race tests, fuzzing and honest docs [HIGH-3]
9. Смержи в main.

Критерий успеха: после этого шага проект можно честно показывать как серьёзную
launch-ready монету, а не как «демо-ноду с красивым README».
